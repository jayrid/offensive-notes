1. **Disable Unnecessary Information Disclosure**
   - Why: The attacker used `/phpinfo.php` and directory listings to gather intelligence on the environment.
   - How: Remove `/phpinfo.php` and disable directory indexing in the Apache configuration (`Options -Indexes`).

2. **Implement a Web Application Firewall (WAF)**
   - Why: A WAF could have blocked the characteristic injection payloads (e.g., `UNION SELECT`, `|id`) observed during the attack.
   - How: Deploy ModSecurity or a similar WAF with the OWASP Core Rule Set (CRS) to detect and block common web attack patterns.

3. **Harden Session Management**
   - Why: Reflected XSS was used to expose session cookies, which were missing the `HttpOnly` flag.
   - How: Configure the application to set the `HttpOnly` and `Secure` flags on all session cookies to mitigate the risk of cookie theft via XSS.

## Detection Rules Recommended
Based on observed IOCs, create monitoring for:
- **Network:**
    - Alert on HTTP GET/POST requests containing `UNION+SELECT`, `ORDER+BY`, or common database schema names (e.g., `information_schema`).
    - Alert on HTTP POST requests to `/vulnerabilities/exec/` containing shell operators like `|`, `;`, `&`, or `$()`.
- **Host:**
    - Monitor web server logs for status code 200 on sensitive paths like `/config/`, `/admin/`, and for files with `.bak` or `.sql` extensions.
    - Alert on system-level execution of `id`, `cat /etc/passwd`, or `whoami` by the `www-data` user.
- **Behavioral:**
    - Alert on high-frequency directory enumeration followed by successful access to a sensitive file or endpoint.
